The CDC Problem

When two parts of a design run on different clocks, passing data between them violates setup/hold timing.

Single-bit CDC: 2-FF synchronizer (Day 5 revisited)

Multi-bit CDC: Gray code for counters/pointers, async FIFO for data

If you have one clock, you have zero CDC problems. Every additional clock multiplies complexity.
Clock domain crossing: different clocks can violate setup/hold. Single-bit: 2-FF sync. Multi-bit: Gray code or async FIFO. One clock = zero problems.

Single-Bit CDC: 2-FF Synchronizer

module cdc_single_bit (
    input  wire i_clk_dst,      // destination clock
    input  wire i_signal_src,   // from source clock domain
    output wire o_signal_dst    // synchronized to destination
);
    reg r_meta, r_sync;

    always @(posedge i_clk_dst) begin
        r_meta <= i_signal_src;
        r_sync <= r_meta;
    end

    assign o_signal_dst = r_sync;
endmodule
Same synchronizer from Day 5, now applied to clock domain crossing. Cost: 2 cycles of latency in the destination domain.
Same 2-FF synchronizer from Day 5, applied to CDC. The first FF resolves metastability; the second ensures a clean value. 2 cycles latency in destination domain.

Multi-Bit CDC: Gray Code

// Binary to Gray (only 1 bit changes per increment)
assign gray = binary ^ (binary >> 1);

// Gray to Binary
assign binary[N-1] = gray[N-1];
generate
    for (g = N-2; g >= 0; g = g - 1)
        assign binary[g] = binary[g+1] ^ gray[g];
endgenerate

Safe for counters/pointers — only 1 bit changes per increment, so the receiver never sees a corrupted intermediate value.

Arbitrary data needs an async FIFO (dual-clock FIFO with Gray-coded pointers).

Gray code: only one bit changes per increment. Safe for counters crossing domains. Arbitrary multi-bit data needs an async FIFO — a standard but complex design pattern.

Pre-Class Self-Check

Q1: Your critical path is 35 ns at 25 MHz (40 ns period). What’s the slack? Is timing met? Fmax?

Slack = 40 − 35 = 5 ns (positive → timing met). Fmax ≈ 1/35 ns ≈ 28.6 MHz.

Q2: What nextpnr flag enables timing analysis?

--freq 25 (specifies target clock in MHz). Without it, no timing check occurs.

Pre-Class Self-Check (cont.)

Q3: What tool calculates iCE40 PLL parameters? What signal must you wait for?

icepll -i 25 -o 50 calculates DIVR/DIVF/DIVQ. Wait for the LOCK signal to go high before using the PLL clock.

Q4: How do you safely pass a multi-bit counter across clock domains?

Convert to Gray code (only 1 bit changes per increment), synchronize with 2-FF, then convert back. Prevents corrupted intermediate values.
