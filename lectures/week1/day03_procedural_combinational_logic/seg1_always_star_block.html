The always @(*) Block

Why Procedural Blocks?

assign works for simple expressions, but as logic gets complex — multi-way decisions, large lookup tables — it becomes unwieldy.

The always block gives us if/else, case, and other procedural constructs.

We need procedural blocks because assign gets unwieldy for complex logic. The always block gives us if/else and case statements.

Sensitivity Lists

// Manual list — DON'T do this
always @(a or b or sel)
    if (sel) y = a; else y = b;
// Wildcard — ALWAYS do this for combinational
always @(*)
    if (sel) y = a; else y = b;
If you forget a signal in a manual list: simulation won't update when that signal changes, but synthesis WILL. Your simulation lies to you. Use @(*) — no exceptions.
Sensitivity lists control when an always block re-evaluates. Manual lists are error-prone — forget one signal and you get a sim-synth mismatch. Always use the wildcard @star for combinational logic. No exceptions.

reg Inside always — Again, Not a Register

reg [3:0] r_result;  // declared as reg...
always @(*)          // ...but combinational (no clock)!
    r_result = a + b;  // → synthesizes to an ADDER, not a register
Rule: wire with assign. reg inside always.
Whether it's a register depends on the sensitivity list, not the keyword.
Signals assigned inside an always block must be declared as reg. But reg does not mean register when used with always-star. It synthesizes to combinational logic. The keyword is misleading — it just means 'can be assigned procedurally.'

Key Takeaways

① Use always @(*) for all combinational procedural blocks.

② Never use manual sensitivity lists — sim/synth mismatch risk.

③ reg inside always @(*) = combinational, not a register.

④ Use begin/end for multi-statement blocks.
