Two Assignment Operators

= Blocking

Evaluate and assign immediately.

Use in: always @(*)

<= Nonblocking

Evaluate all RHS first, then update all LHS simultaneously.

Use in: always @(posedge clk)

= in always @(*).   <= in always @(posedge clk).
Never mix them in the same block.
Blocking equals for combinational. Nonblocking less-than-equals for sequential. Never mix. We prove why on Day 4 with timing diagrams.

Why It Matters (Preview)

// WRONG: blocking in sequential block
always @(posedge clk) begin
    b = a;    // b gets a's value NOW
    c = b;    // c gets b's NEW value (= a!) — WRONG
end
// CORRECT: nonblocking in sequential block
always @(posedge clk) begin
    b <= a;   // scheduled: b will get a's current value
    c <= b;   // scheduled: c will get b's current value
end          // both update simultaneously at end of timestep
With <=, both stages see pre-update values. This models real flip-flop behavior where all FFs capture simultaneously on the clock edge.
Blocking in sequential: b gets a immediately, then c gets the new b — both get the same value. Nonblocking: both see pre-update values, modeling real flip-flop behavior.

Key Takeaways

① = (blocking) for combinational always @(*).

② <= (nonblocking) for sequential always @(posedge clk).

③ Mixing them causes race conditions — Day 4 will prove it.

Pre-Class Self-Check

Q1: Why must you use @(*) instead of a manual sensitivity list?

Manual lists risk sim/synth mismatch. @(*) automatically includes all signals read inside the block.

Q2: What causes an unintentional latch?

Not assigning a signal in every possible path through always @(*). Example: if (sel) y = a; with no else.

Pre-Class Self-Check (cont.)

Q3: Name three techniques to prevent latch inference.

1. Default assignment at top. 2. Complete if/else chains. 3. default in case statements.

Q4: When do you use = vs <=?

= in combinational always @(*). <= in sequential always @(posedge clk). Never mix.
