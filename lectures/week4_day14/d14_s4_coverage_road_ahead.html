Functional Coverage: Are You Done Testing?

Simulation can show bugs exist. It can never prove they don't. Coverage measures how much of the design space you've exercised.

// SystemVerilog coverage constructs (conceptual — not supported in iverilog)
covergroup alu_cov @(posedge clk);
    cp_opcode: coverpoint opcode {
        bins add = {3'b000}; bins sub = {3'b001};
        bins bw_and = {3'b010}; bins bw_or = {3'b011};
    }
    cp_a_range: coverpoint a {
        bins zero = {0}; bins mid = {[1:254]}; bins max = {255};
    }
    cx_full: cross cp_opcode, cp_a_range;  // 4 × 3 = 12 bins
endgroup
Tool note: Icarus Verilog doesn't support covergroup. In today's lab, you'll analyze coverage manually — which is the harder and more valuable skill.
Coverage is how you answer "am I done testing?" Iverilog can't do automatic coverage, so you'll do it manually — which actually teaches you to think about the design space, not just collect numbers.

Where These Skills Lead

UVM (Universal Verification Methodology): Your constraint specs → UVM sequences. Your AI-generated TBs → UVM test library. Your assertions → SVA property specifications.

Formal verification: Mathematically prove properties hold for all possible inputs. Tools: SymbiYosys (open-source), JasperGold, VC Formal (commercial).

AI in professional V&V: Industry uses AI for TB scaffolding, regression debugging, coverage analysis, and assertion generation — the same workflow you've practiced, at scale.

Interfaces & modport (brief): SV interface bundles signals; modport defines views. Reduces port-map boilerplate. Stretch exercise in today's lab.

Quick tour of where the skills you've built go next. UVM, formal, AI-assisted V&V, interfaces — all build directly on the foundation from this course. The department's V&V course covers UVM and formal in depth.

Pre-Class Self-Check

Q1: What does assert (o_tx == 1'b1) else $error("TX not idle"); do that if (!o_tx) $display("Error") doesn't?

Assertions are standardized (tools count/filter/report uniformly), have severity levels, can be globally enabled/disabled, and are recognized by formal verification tools. if-statements provide none of these benefits.

Q2: Write a constraint spec (2–3 lines) for testing a UART TX module with AI.

"Test UART TX at 115200 baud with $urandom_range for data bytes. Send at least 20 frames including 0x00, 0xFF, and alternating 0xAA/0x55. Verify each frame has correct start bit (0), 8 data bits LSB-first, and stop bit (1). Self-checking: compare TX output timing against CLKS_PER_BIT."

Pre-Class Self-Check (cont.)

Q3: Your PPA table shows adding parity costs 6 LUTs. Is this "expensive"? How do you decide?

6 LUTs = 0.5% of the iCE40 HX1K (1,280 total). On this FPGA, it's negligible. But "expensive" depends on context: if you're already at 90% utilization, every LUT matters. The analysis paragraph should state the absolute cost AND the relative cost.

Q4: Name the three FPGA PPA proxies and their measurement tools.

Performance → Fmax (nextpnr --freq). Area → LUTs/FFs/EBR (yosys stat). Power → toggle rate × capacitance (conceptual on iCE40). These proxy the ASIC metrics: critical path delay, gate count/die area, and dynamic+static power.
