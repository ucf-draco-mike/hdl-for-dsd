The Verification Productivity Stack

Level 1: Manual directed tests — you write every stimulus value (Day 4)

Level 2: Self-checking TBs — automated pass/fail, expected vs. actual (Day 6)

Level 3: AI-scaffolded TBs — AI generates boilerplate; you review + correct (Days 6, 8, 12)

Level 4: Assertion-enhanced — executable specs catch bugs at source (today, Segment 1)

Level 5: Constraint-based + coverage-driven — define the input space, measure what you've tested

You've climbed from Level 1 to Level 4 over the past two weeks. Today's lab takes you to Level 5.
This is the verification maturity scale. Students are now at Level 4. The jump to Level 5 is about defining the input space formally (constraints) and measuring what you've tested (coverage).

Constraint-Based Stimulus Generation

Instead of listing specific test values, define the legal input space:

// Not this (directed):
a = 8'd15; b = 8'd7; opcode = 3'b000;  // one specific test

// This (constrained random):
a = $urandom_range(0, 2**WIDTH - 1);
b = $urandom_range(0, 2**WIDTH - 1);
opcode = $urandom_range(0, 4);         // all valid opcodes

Bounded random: $urandom_range(min, max) generates values within legal ranges.

Constraints: Relationships between signals — "if opcode is ADD, ensure at least 10 overflow cases."

Directed tests check specific cases you thought of. Constrained random explores the space you defined. It finds bugs you didn't think to test for — the ones that matter most.

Writing a Constraint Specification

The spec is what you give the AI. Be precise:

Module: alu (input [7:0] a, b; input [2:0] opcode; output [7:0] result)
Operations: ADD(000), SUB(001), AND(010), OR(011), XOR(100)

Test at least 100 random cases per opcode.
For ADD/SUB: ensure at least 10 cases each of:
  - no overflow, overflow, zero result, max result
For AND/OR/XOR: include all-zeros, all-ones, alternating patterns.
Use $urandom_range() for stimulus.
Self-checking: compute expected in the testbench, compare.
Report per-opcode pass/fail counts at end.
Key skill: A vague spec → vague TB. A precise spec → a TB you can evaluate. The quality of the spec determines the quality of the AI output.
This is the format for your lab exercise today. Write the spec, give it to the AI, evaluate what comes back. The spec is the contract — it's what lets you judge whether the AI delivered or not.

Key Takeaways

① Define the input space with constraints, not individual vectors.

② A precise constraint spec → a reviewable AI output. Vague spec → garbage.

③ $urandom_range() for bounded random. Count your coverage.

④ The evaluation skill transfers directly to industry verification workflows.
