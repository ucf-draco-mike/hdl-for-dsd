The Problem With Manual Inspection

✗ Miss subtle bugs (off-by-one, wrong bit, glitch)

✗ Doesn’t scale — can’t visually verify 1000 test vectors

✗ Not repeatable or automatable

The testbench must check itself. If it doesn’t print PASS or FAIL, you’re doing it wrong.
Looking at waveforms doesn't scale. You miss bugs, it's not repeatable, can't automate. The testbench must check itself.

Automated Checking Pattern

integer test_count = 0, fail_count = 0;

    // Test: ADD 3 + 5 = 8
    a = 4'd3; b = 4'd5; opcode = 2'b00;
    #100;

    test_count = test_count + 1;
    if (result !== 4'd8) begin
        $display("FAIL: ADD 3+5: expected 8, got %d", result);
        fail_count = fail_count + 1;
    end else
        $display("PASS: ADD 3+5 = %d", result);
!== not != — case inequality handles x/z correctly. x != 0 → x (unknown!). x !== 0 → 1 (definite mismatch).
Track test and fail counts. Use case inequality — three-equals handles x and z. Regular inequality returns unknown when either side is x.

Summary Report

// At end of simulation
$display("\n=== TEST SUMMARY ===");
$display("Tests run:    %0d", test_count);
$display("Tests passed: %0d", test_count - fail_count);
$display("Tests failed: %0d", fail_count);
if (fail_count == 0)
    $display("*** ALL TESTS PASSED ***");
else
    $display("*** %0d FAILURES ***", fail_count);
$finish;
This is your standard from now on. Every testbench prints a summary.
Every testbench ends with a summary. Tests run, passed, failed. This is your standard going forward.

Key Takeaways

① Self-checking: if (actual !== expected) — automate it.

② Use ===/!== (case equality) — handles x/z correctly.

③ Track test_count and fail_count. Print a summary.

④ This pattern applies to every testbench going forward.
