A testbench is a module with no ports that drives your DUT

You've been running testbenches since Day 1. Every time you typed make-sim and saw PASS or FAIL, a testbench was doing the work. This week, you learn to write them yourself — same patterns you've been relying on, but now you build them from scratch. A testbench is a port-less module that exercises your design under test. Let's look at the anatomy.

Testbench Anatomy

`timescale 1ns / 1ps
module tb_my_design;
    reg        clk, reset;          // YOU drive DUT inputs → reg
    reg  [3:0] a, b;
    wire [3:0] result;              // DUT drives outputs → wire

    my_design uut (                 // Unit Under Test
        .i_clk(clk), .i_reset(reset),
        .i_a(a), .i_b(b), .o_result(result)
    );

    always #20 clk = ~clk;         // 25 MHz clock (40ns period)

    initial begin
        $dumpfile("tb_my_design.vcd");
        $dumpvars(0, tb_my_design);
        clk = 0; reset = 1; a = 0; b = 0;
        #100; reset = 0;
        // ... stimulus ...
        #1000; $finish;
    end
endmodule
Testbench template: timescale, reg for DUT inputs, wire for outputs, instantiate DUT, clock generator, dumpfile/dumpvars, initial block with stimulus, finish.

Key Takeaways

① Testbench = no-port module. reg for DUT inputs, wire for outputs.

② `timescale, clock gen, $dumpfile/$dumpvars, $finish — memorize the template.

③ Every design gets a testbench from today forward.

④ Simulate first. Synthesize second. Program last.
