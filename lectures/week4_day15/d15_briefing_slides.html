⚠️ The #1 Failure Mode

❌ The “Big Bang”

Write the whole design top-to-bottom. Instantiate every module. Run full simulation. Try to program. Everything fails at once. Stare at 400 lines of Verilog across 6 files. Three hours evaporate. Panic sets in. Final hour is frantic fixes, one of which breaks another. Submit broken.

✓ Incremental Integration

Bring up one module at a time. Each increment is testable in isolation before adding the next. Break = back out one step, fix, proceed. At any point, you have a working partial system. If time runs out, you submit something working — just fewer features.

The receipt: Industry postmortems agree — the most common cause of missed deadlines is monolithic integration. “I wrote the whole thing and then tried to test it” is the #1 antipattern. Every senior engineer learns this the hard way once. Today you learn it vicariously.
The monolith is seductive — “I'll write it all, then test.” It never works. Incremental is harder to start but faster to finish.

Incremental Integration

The disciplined sequence for today:

  1. Step 1 — Blink. Top-level module with one LED + clock + reset. Synthesize. Program. Verify LED blinks. This proves your flow (make, program, hardware) works. 10 minutes max.
  2. Step 2 — Skeleton top module. Add empty instantiations of every submodule you'll need. No logic inside. Port lists wired. Compiles + passes simple smoke test. ~15 minutes.
  3. Step 3 — Bring up one submodule. Fill in one module (start with the simplest — usually the clock divider or reset synchronizer). Add targeted testbench. Sim passes → program → verify on board. ~20 minutes per submodule.
  4. Step 4 — Integrate pairs. Once two modules work individually, check they work composed. Backpressure? Clock alignment? Reset ordering? ~15 min per pair.
  5. Step 5 — Full system. Everything wired, everything tested, now run end-to-end. Most surprises are already caught. ~30 minutes.
Key property: after each step you have a working system — even if it does less than the final spec. Submit-at-any-time always beats monolithic-at-the-last-minute.
Five steps. Each increment leaves you with a working (if simpler) system. Never leave yourself in a state where nothing works.

Step 2: Skeleton Top Module

module capstone_top (
    input  logic i_clk,
    input  logic i_switch1, i_switch2, i_switch3, i_switch4,
    output logic o_led1, o_led2, o_led3, o_led4,
    output logic o_uart_tx
);
    // Reset synchronizer (Week 2) — bring up first
    logic r_rst_sync;
    reset_sync u_rst   (.i_clk, .i_rst_async(i_switch1), .o_rst_sync(r_rst_sync));

    // UART TX (Week 3) — add after reset works
    logic       w_tx_valid, w_tx_busy;
    logic [7:0] w_tx_data;
    uart_tx #(.CLKS_PER_BIT(217)) u_tx (
        .i_clk, .i_rst(r_rst_sync),
        .i_valid(w_tx_valid), .i_data(w_tx_data),
        .o_busy(w_tx_busy),   .o_tx_line(o_uart_tx)
    );

    // FSM controller (new for capstone) — add last
    capstone_fsm u_fsm (
        .i_clk, .i_rst(r_rst_sync),
        .i_sw({i_switch4, i_switch3, i_switch2}),
        .o_led({o_led4, o_led3, o_led2, o_led1}),
        .o_tx_valid(w_tx_valid), .o_tx_data(w_tx_data),
        .i_tx_busy(w_tx_busy)
    );
endmodule
Observe: the top reads like a wiring diagram. All the work is in the sub-modules; the top just composes them. Echoes Topic 12's “wrapper smaller than sub-modules” rule. Each sub-module can be brought up one at a time.
Skeleton first, flesh later. The top-level is wire routing; each sub-module is testable in isolation. Incremental fill-in.

Common Late-Stage Panics

SymptomLikely causeFix
Nothing happens on boardReset polarity or stuck in resetCheck reset synchronizer; probe reset with LED
LEDs flicker randomlyMetastability on input, missing sync_2ffAdd 2-flop sync on any async input
UART outputs garbageWrong CLKS_PER_BIT or clock mismatchVerify clock freq; recompute for actual baud
FSM stuck in one stateTransition condition never metAdd debug LED per state; run slowly
Fits in sim but not on boardUnconstrained I/O, missing pin mappingCheck .pcf file; every external pin listed
Timing violation in reportCombinational path too longAdd pipeline register; check Fmax report
Compiles once, fails next timeLeftover files, stale cachemake clean && make always
Golden rule of today: when something breaks, don't randomly edit. Revert to the last working state. Change one thing. Re-test. The discipline feels slow; it's actually fast.
Panic moves are random edits. Discipline moves are “revert, change one thing, re-test.” Students who stay disciplined under pressure finish. Students who panic don't.

UART Printf Debugging

Your UART TX is now a debug instrument. Wire it up so the FSM can send a byte per state transition to your laptop terminal. Silicon's closest thing to printf.

// In FSM, on each state change:
always_ff @(posedge clk) begin
    if (r_state != r_prev_state) begin
        o_tx_valid <= 1'b1;
        case (r_state)
            S_IDLE: o_tx_data <= 8'h49;  // 'I'
            S_RUN:  o_tx_data <= 8'h52;  // 'R'
            S_DONE: o_tx_data <= 8'h44;  // 'D'
            S_ERR:  o_tx_data <= 8'h45;  // 'E'
        endcase
    end else o_tx_valid <= 1'b0;
    r_prev_state <= r_state;
end
Power: open minicom, press buttons on the Go Board, watch the FSM's trace stream to your terminal. “I → R → D” tells you the FSM progressed through states you expected. “I → R → R → E” tells you it went to error — and you know exactly where. This is the single highest-yield debug technique you have today.
UART printf is the closest thing to gdb you have on an FPGA. Students often don't think to use it; it transforms debug from guessing to observing.

What You're Submitting

  • RTL files — your top-level + any new sub-modules. Existing library modules referenced, not duplicated.
  • Testbench — at least one self-checking testbench showing end-to-end function.
  • Build log — make stat output showing LUT/DFF usage and make place output showing Fmax. Annotate which metric was your binding constraint.
  • Brief reflection — 1-2 paragraphs: what worked, what broke, what you'd change given another hour.
Grading philosophy: graded on working-ness, not ambition. A smaller design that fully works beats a grander one that half-works. Integration discipline over feature count.
Explicit grading philosophy — discipline > ambition. Students often misallocate effort chasing features when basic function is uncertain. Lead with working.

🤖 AI During Build Day

Today's AI rules (re-statement of Topic 14.2 principles):

✓ Use AI for

  • Boilerplate testbenches for modules you've already defined
  • Syntax reminders (“what's the SVA syntax for N-cycle delay?”)
  • Error message diagnosis (“iverilog says X at line Y, what does that mean?”)
  • Code review (“look at my FSM and tell me what might go wrong”)

❌ Don't use AI for

  • Writing the whole design from the spec — that's your work
  • “Does this work on the Go Board?” — AI doesn't know; check the hardware
  • Pretending its passing testbench means anything without you running it
  • Substitute for reading your synthesis report
AI rules hold on build day. Use it to accelerate mechanics; own the intent and the observation.

Key Reminders

① Incremental > monolithic. Small working system beats big broken one.

② Revert on failure. Change one thing. Re-test. Discipline wins.

③ UART printf is your on-silicon debugger. Use it early, not late.

④ 30-minute checkpoint: “what's my last known-good state?”

At any point in the day, you should have a working system. Today's success is not feature count — it's the discipline that gets you to working silicon.
