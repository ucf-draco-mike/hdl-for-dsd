The #1 Failure Mode

Big Bang Integration

Write all modules → connect everything at once → hope it works

It never works.

Wall of errors. No idea which module is broken. No path forward.

Big bang integration is the number one failure mode on project day. The fix is incremental integration.

Incremental Integration

Step 1: Verify module library — all testbenches pass

Step 2: Skeleton top module + heartbeat LED → synthesize immediately

Step 3: Add ONE module. Synthesize. Verify on hardware. Repeat.

Step 4: UART printf debugging for hardware-only issues

Each step produces a working system. If something breaks, it was the last thing you added.
Four steps. The key insight: each step produces a working system. When something breaks, you know exactly what caused it.

Step 2: Skeleton Top Module

module top_project (
    input  logic i_clk,
    input  logic i_switch1, i_switch2, i_switch3, i_switch4,
    input  logic i_uart_rx,
    output logic o_uart_tx,
    output logic o_led1, o_led2, o_led3, o_led4,
    // ... 7-seg ports ...
);
    // Heartbeat — proves FPGA is configured and running
    logic [23:0] heartbeat;
    always_ff @(posedge i_clk)
        heartbeat <= heartbeat + 1;
    assign o_led4 = ~heartbeat[23];

    // Stub everything else
    assign o_uart_tx = 1'b1;  // idle high
    assign {o_led1, o_led2, o_led3} = 3'b111; // off
endmodule
Synthesize and program this first. Heartbeat blinks? Toolchain works. Now add modules one at a time.
Skeleton top with heartbeat. Synthesize immediately. If the LED blinks, your toolchain and FPGA are working before you add any complexity.

Common Late-Stage Panics

“Works in sim, not on hardware” → Check pin mappings, debounce, clock frequency params. Add debug LEDs.

“I can’t fit it” → Check stat output. Often caused by synthesis inferring unexpected logic (e.g., huge multipliers).

“Timing fails” → Usually fine at 25 MHz. Check for long combinational chains (wide muxes, deep priority encoders).

“I haven’t started” → Simplest possible demo: heartbeat + one button-driven action. Start there.

Four common panics and their fixes. The key message: there's always a path forward, even if it means reducing scope.

UART Printf Debugging

// Hardware printf: send a status byte via UART
always_ff @(posedge i_clk) begin
    if (w_debug_trigger && !w_uart_busy) begin
        uart_data  <= {"S", state};  // or a debug code
        uart_valid <= 1'b1;
    end else
        uart_valid <= 1'b0;
end
Your UART TX module becomes a hardware debugger. Send state codes, counter values, or flag bytes to the PC terminal.
UART printf debugging: reuse your UART TX to send debug data to the PC terminal. This is a real industry technique — hardware printf.
