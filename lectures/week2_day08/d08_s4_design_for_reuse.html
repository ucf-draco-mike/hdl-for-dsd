Reuse Checklist

☑ Parameterized — widths, thresholds, counts are parameters

☑ Self-contained — no dependency on external magic numbers

☑ Tested — self-checking testbench at multiple configurations

☑ Documented — header comment with description, ports, parameters

☑ Named — clear, descriptive file and module names

A module you can’t reuse without reading its source is not reusable.
Reuse checklist: parameterized, self-contained, tested, documented, named. If you have to read the source to use it, it's not truly reusable.

Pre-Class Self-Check

Q1: What is the difference between parameter and localparam?

parameter can be overridden at instantiation. localparam is internal/derived and cannot be overridden — use for computed constants.

Q2: Is generate for a runtime loop?

No! It runs at elaboration (compile time) and creates N physical copies of the hardware. It is NOT a sequential loop.

Pre-Class Self-Check (cont.)

Q3: What does $clog2(1000) return?

10 — because ceil(log₂(1000)) = 10. You need 10 bits to represent values 0 through 999.

Q4: Why name generate blocks (begin : gen_name)?

Named blocks create hierarchical paths visible in simulation and synthesis reports. Debugging gen_debounce[2].db is much easier than anonymous generated instances.
