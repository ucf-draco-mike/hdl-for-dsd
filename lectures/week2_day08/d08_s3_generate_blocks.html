Video 3 of 4. Compile-time hardware replication. One block of code → N physical instances on the chip.

🌍 Where This Lives

In Industry

Every wide bus in a modern SoC is a generate array. 64-bit adders: 64 full-adder cells generated. 8-way set-associative caches: 8 way modules generated. SIMD lanes (NEON, AVX): N parallel ALUs generated. The CPU pipeline is often N parallel execution units generated. A parameterized generate block is how you scale a design from a 4-core chip to a 64-core chip by changing one number.

In This Course

Your Go Board has 4 buttons — you'll generate 4 debouncers instead of copy-pasting. Your Day 11 parallel-to-serial converter generates per-bit logic. Day 12 SPI uses it for variable-width word sizes. Real products scale through this feature.

Generate is how you scale. Without it, you copy-paste — which doesn't scale, doesn't refactor, doesn't read well. With it, one parameter resizes the whole design.

⚠️ generate for Is Not a Software Loop

❌ Wrong Model

“generate for (i=0; i<N; i++) is a for-loop. It executes N times during simulation, one iteration per clock.”

✓ Right Model

A generate for is not a runtime loop. It is compile-time replication. The synthesizer unrolls the loop at compile time, producing N physical copies of the hardware it contains. After synthesis, no loop exists. Only N parallel circuits running simultaneously.

The receipt: generate for i=0..7 with a full-adder inside = 8 physical full-adders, all active in parallel. Like having typed 8 instantiations. Yosys literally unrolls before synthesis.
The biggest reframe of the video. Generate is unrolled at compile time. After synth, the loop is gone — only the N copies remain. This is fundamental to understanding what you're building.

👁️ I Do — Array of Debouncers

module button_array #(
    parameter N = 4                 // number of buttons
) (
    input  wire         i_clk, i_reset,
    input  wire [N-1:0] i_buttons,  // raw noisy buttons
    output wire [N-1:0] o_clean     // debounced clean signals
);
    genvar gi;                       // generate-index variable
    generate
        for (gi = 0; gi < N; gi = gi + 1) begin : g_btn
            debounce #(.CLKS_STABLE(500_000)) u_deb (
                .i_clk   (i_clk),
                .i_reset (i_reset),
                .i_noisy (i_buttons[gi]),
                .o_clean (o_clean[gi])
            );
        end
    endgenerate
endmodule
My thinking: Three idioms. (1) genvar gi declares a compile-time loop index. (2) begin : g_btn names the generate scope — required, and shows up in waveforms as dut.g_btn[0].u_deb. (3) The inner code looks like one debouncer instantiation; the generator creates N of them.
Standard pattern for “array of this thing.” Named scope is mandatory and shows up in waveforms — very important for debug navigation.

🤝 We Do — generate if for Conditional Hardware

module configurable_pipe #(
    parameter USE_SYNCHRONIZER = 1   // compile-time switch
) (
    input wire i_clk, i_in, output wire o_out
);
    generate
        if (USE_SYNCHRONIZER) begin : g_sync
            reg r_meta, r_stable;
            always @(posedge i_clk) begin
                r_meta   <= i_in;
                r_stable <= r_meta;
            end
            assign o_out = r_stable;
        end else begin : g_nosync
            assign o_out = i_in;     // direct pass-through — no flops at all
        end
    endgenerate
endmodule
Together: generate if includes hardware conditionally based on parameter values. Setting USE_SYNCHRONIZER=0 means the synthesizer builds no synchronizer — not even a wire. This is how you optimize instances that don't need certain features: internal signals skip the synchronizer, external signals get one.
Generate if = compile-time option. Unlike an always block with a variable condition, the unused branch produces zero silicon — not “inactive” hardware, zero hardware. Great for feature toggles.

🧪 You Do — Generate an N-Bit Ripple Adder

Write a generate block that builds an N-bit ripple-carry adder out of N full-adder instances. Full adder has inputs a, b, cin and outputs sum, cout.

Sketch: 
wire [N:0] carry;
assign carry[0] = cin;
genvar gi;
generate for (gi = 0; gi < N; gi = gi + 1) begin : g_fa
    full_adder u_fa (
        .a(a[gi]), .b(b[gi]), .cin(carry[gi]),
        .sum(sum[gi]), .cout(carry[gi+1])
    );
end endgenerate
assign cout = carry[N];
The carry wire of width N+1 chains the full-adders — element gi+1 reads what element gi wrote. This is the classic pattern for bit-serial structures.
Ripple carry adder is the canonical generate-for example. Notice the carry chain wire has width N+1 to accommodate the final cout. Students often get the carry wire width wrong by one.

▸ KEY OBSERVATION

One Verilog file, one parameter change, three different-sized designs. The Verilog doesn't grow — only the gate count does. This is what makes generate a force multiplier.

[Terminal] Same file, different N, different cell counts. Perfectly linear scaling because each debouncer is independent. This is the magic of generate: your source stays the same size, your silicon scales.

🔧 What Did the Tool Build?

$ yosys -p "read_verilog ... button_array.v; chparam -set N 4 button_array; \
           synth_ice40; stat"

=== button_array ===   # N=4
   Number of cells:  160   (= 4 × ~40-cell debouncer)
   Contains sub-instances:
     g_btn[0].u_deb   (debounce)
     g_btn[1].u_deb   (debounce)
     g_btn[2].u_deb   (debounce)
     g_btn[3].u_deb   (debounce)
What to notice: The named generate scope (g_btn) appears in hierarchy output and in GTKWave signal paths. tb.dut.g_btn[2].u_deb.r_count is the counter of the 3rd debouncer. Without that naming, you couldn't navigate generated hardware.
Checkpoint: Generate + parameter is the “infinite IP” pattern. One source file serves 1, 10, 1000 instances — never rewrite, just re-instantiate.
Named scope = debuggable generate. Waveform paths include the generate index. Without naming, you can't tell one debouncer's signals from another's.

BEFORE

AFTER

Strong AI uses genvar, named scopes. Weak AI forgets the named scope.

TAKEAWAY

Require named scopes in your prompt. AI often forgets without explicit ask.

AI is competent at generate-for, sometimes forgets named scopes. Always ask for them explicitly.

Key Takeaways

① generate for = compile-time hardware replication, not a runtime loop.

② generate if = conditional hardware; unused branch produces zero silicon.

③ Always use genvar and named scopes (begin : g_foo).

④ Combined with parameters, generate scales a design by changing one number.

If you're about to copy-paste a module N times, use generate instead.
