Hierarchy Manages Complexity

    top_button_counter
    ├── debounce (switch 1 — reset)
    ├── debounce (switch 2 — count)
    ├── edge_detector (count press)
    ├── counter_mod_n (4-bit hex counter)
    ├── hex_to_7seg (display decoder)
    └── heartbeat_blinker (LED1)

Already 6+ modules in a simple design. A UART controller will have twice as many.

Good hierarchy: each module is small enough to understand, test, and reuse independently.
By Day 8, designs are complex. Good hierarchy makes each piece small enough to understand, test, and reuse independently.

Naming Conventions for Hierarchy

// Descriptive instance names (not u1, u2!)
debounce #(.CLKS_TO_STABLE(250_000)) debounce_sw1 (
    .i_clk    (i_clk),
    .i_bouncy (i_switch1),
    .o_clean  (w_sw1_clean)
);

// Internal wires named by purpose
wire w_sw1_clean;    // ✓ describes what it carries
wire w_count_pulse;  // ✓ describes its role
wire w_data;         // ✗ too vague
Rules: Descriptive instance names. Always named port connections. Internal wires describe purpose.
Descriptive instance names, named port connections, descriptive wire names. Never u1/u2, never positional ports, never w_data.

Key Takeaways

① Hierarchy = each module independently understandable and testable.

② Descriptive instance names: debounce_sw1, not u1.

③ Always named port connections — never positional.

④ Design top-down (architecture), implement bottom-up (test small first).
