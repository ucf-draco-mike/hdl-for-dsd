🌍 Where This Lives

In Industry

The FSM + datapath decomposition is the canonical shape of every digital controller. Every USB device splits “enumeration state machine + data pipeline.” Every SPI controller splits “transaction FSM + shift registers.” Every PCIe endpoint, every Ethernet MAC, every SATA controller — all FSM-plus-datapath. Mastering this decomposition means you can design any protocol endpoint.

FSM + datapath is the canonical protocol controller shape. Separating control from data is a universal design pattern.

⚠️ Don't Try to Do It All in One Block

❌ Wrong Model

“I'll write one big always block with nested ifs that handle the start bit, shift out data, and signal stop bit. One block = clean code.”

✓ Right Model

Separate control (“what should the TX line do right now?” — an FSM) from datapath (“what is the current bit value?” — a shift register + counter). The FSM tells the datapath when to load, shift, and idle. The datapath holds the byte and produces the serial output. Each block stays under 50 lines, each testable independently.

The receipt: One-block UART TXs look clever but accumulate bugs with every edit. FSM + datapath UARTs are textbook-clean and maintainable for decades.
FSM+datapath separation is a design discipline. It keeps each piece small, testable, and maintainable. Monolithic controllers are a smell.

🤝 We Do — The FSM States

StateOutputDurationExit condition
S_IDLETX=1 (idle high)until requesti_valid high → S_START
S_STARTTX=0 (start bit)1 bit timebaud tick → S_DATA
S_DATATX=shift_lsb8 bit times8 ticks → S_STOP

Together: Four states, each maps to one part of the UART frame. The FSM advances on each baud tick. In S_DATA, a sub-counter (0..7) counts bits. The FSM stays in S_DATA for 8 baud ticks, then transitions to S_STOP. Clean 3-block template from Week 2.
Four states. Each corresponds to one part of the 10-bit frame. Stay in DATA for 8 bit times. Binary encoding (2 bits of state), 3-block template.

UART TX FSM — State Diagram

Read the picture: Four states, each producing a fixed o_tx value (Moore — outputs depend only on state). The whole frame is just “walk the loop once”: IDLE → START → DATA (×8) → STOP → IDLE. baud_tick is the only thing that advances state; everything between ticks is the bit-time wait. o_busy is high in any non-idle state.
The state diagram makes the “walk once around the loop = transmit one byte” intuition crisp. The DATA self-loop (with the bit counter) is where all 8 data bits live. Compare to the Day 7 traffic-light Moore FSM — same pattern, different domain.

🤝 Valid / Busy Handshake

            ┌──────────────────────────┐
  upstream │                          │  tx line to
  (app) ──▶│ UART TX                  │──────────▶ RS-232 cable
            │                          │
   i_data  │  .i_data[7:0]            │  o_tx
   i_valid │  .i_valid                │
   o_busy  │  .o_busy                 │
            └──────────────────────────┘

  Sequence:
    1. App puts byte on i_data, raises i_valid.
    2. TX sees i_valid && !o_busy → captures byte, starts transmission.
    3. TX raises o_busy for duration of 10-bit frame.
    4. App sees o_busy high → does not submit a new byte yet.
    5. TX returns to idle → o_busy drops → app can send next byte.
Together: This is valid/busy (also called “ready/valid” or AXI-Stream handshake). It's the universal producer/consumer protocol. Upstream says “I have data” (valid); downstream says “I'm busy” (busy). Byte transfers when valid && !busy. Clean, standardized, composable.
Valid/busy is the universal handshake. Same as AXI-Stream, APB, many others. Producer says “I have data,” consumer says “I'm busy.” Standard interface.

🧪 You Do — Count Bit Times

Starting from S_IDLE, how many baud ticks between i_valid assertion and the TX line returning to idle?

Answer: 10 ticks — 1 start + 8 data + 1 stop. That's the “byte time” from Video 1. At 115200 baud, 10 × 8.68 µs = 86.8 µs per byte. At 9600 baud, ~1 ms per byte. Upstream throughput is capped by 1 / byte_time.
Followup: If you need to transmit back-to-back without idle gaps, you can pipeline: accept the next byte during the S_STOP state. But a standards-compliant UART typically leaves at least 1 bit of idle between bytes.
Byte time = 10 bit times. Throughput = 1/byte_time. Students internalize the time structure.

Key Takeaways

① UART TX = FSM (control) + datapath (shift reg + output mux).

② Four states: IDLE, START, DATA (×8), STOP.

③ Valid/busy handshake is the standard producer/consumer protocol.

④ Every building block exists from Week 2 — this is composition, not invention.

Separate control from data. Compose from known blocks. Test each block alone.
