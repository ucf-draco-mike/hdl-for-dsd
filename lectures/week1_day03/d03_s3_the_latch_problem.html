What Is an Unintentional Latch?

If a signal in always @(*) is not assigned in every path, the synthesizer must preserve its old value.

The only combinational element that “remembers” is a latch.

Latches are almost never what you want. They cause timing hazards, are hard to test, and FPGA tools handle them poorly.
If a signal isn't assigned in every path, synthesizer infers a latch. Latches cause timing hazards and are almost never what you want.

How Latches Are Inferred

always @(*) begin
    if (sel)
        y = a;
    // NO else! When sel=0, y must keep old value → LATCH
end
always @(*) begin
    case (opcode)
        2'b00: result = a + b;
        2'b01: result = a - b;
        // 2'b10, 2'b11 missing → LATCH on result
    endcase
end
Two classic patterns: if without else, and incomplete case. Both cause latch inference.

Three Prevention Techniques

① Default assignment (recommended):

always @(*) begin
    y = 4'b0000;         // assign FIRST, override later
    if (enable) y = data;  // no latch — default covers else
end

② Complete if/else:

always @(*) begin
    if (enable) y = data; else y = 4'b0000;
end

③ default in case:

    case (sel) ... default: y = 4'b0; endcase
Best practice: use ① AND ③. Belt and suspenders.
Three techniques: default assignment at top, complete if/else, default in case. Best practice: use technique 1 AND 3 together.

Key Takeaways

① Incomplete paths in always @(*) → unintentional latch.

② Default assignment at the top of every block = best prevention.

③ Always include default in case, even when enumeration is complete.

④ Yosys warns about latches — read the warnings.
