Build vs. Integrate

Build yourself: when learning, custom requirements, module is small

Integrate IP: complex protocols (USB, Ethernet), vendor primitives (PLL), battle-tested open-source

The skill isn’t just building — it’s knowing when to build vs. when to integrate.
Build when learning or when requirements are custom. Integrate when the protocol is complex or battle-tested IP exists. The skill is knowing which to choose.

IP Integration Checklist

① Read the interface spec — ports, protocols, timing requirements

② Write a wrapper — adapt port names, add synchronizers, isolate

③ Write a testbench for the wrapper — don’t trust, verify

④ Check synthesis results — does it fit? Meet timing?

⑤ Document — version, config params, known limitations, coverage

Integration checklist: read docs, write wrapper, add synchronizers, test the wrapper, check synthesis, document everything.

Pre-Class Self-Check

Q1: Why does UART RX use 16× oversampling?

To find the center of each bit period without a shared clock. 16 samples/bit allows center-aligned sampling (tick 7-8) for maximum noise immunity and baud rate tolerance.

Q2: What is the loopback test and why is it the gold standard?

RX→TX echo: type on PC terminal, see characters echoed back. If echo works, both TX and RX are verified correct. One test validates two modules.

Pre-Class Self-Check (cont.)

Q3: How many wires does SPI need? What does the master provide?

4 wires: SCLK, MOSI, MISO, CS_N. Master provides the clock — no baud rate negotiation needed. Full duplex: data flows both ways simultaneously.

Q4: What’s the first thing on the IP integration checklist?

Read the interface specification: understand ports, protocols, and timing requirements before writing any code.
