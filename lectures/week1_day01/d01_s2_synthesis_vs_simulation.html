One Source File, Two Consumers

module.v — your Verilog source

⚙️ Synthesis

"Turn this into hardware"

Yosys → netlist of real gates

🔍 Simulation

"Show me how it behaves"

Icarus Verilog → waveforms

Both consume the same .v file. They do completely different things with it.
Your Verilog source file has two consumers. Synthesis turns it into hardware — actual gates mapped to an FPGA. Simulation shows you how the design behaves without touching hardware. Both consume the exact same dot-v file. They do completely different things with it.

The Open-Source iCE40 Toolchain

Here's our complete toolchain. On the left, the synthesis path: Yosys synthesizes your Verilog into a netlist of iCE40 primitives. nextpnr places and routes those primitives onto the physical FPGA. icepack generates the bitstream. iceprog loads it onto the board via USB. On the right, the simulation path: Icarus Verilog compiles your Verilog and testbench, vvp runs the simulation, and GTKWave lets you view the waveforms. Same source file, two completely different pipelines.

What Each Path Produces

Synthesis Output

Netlist — a circuit of real gates (LUTs, FFs)

Placement — which physical location on the chip

Routing — which wires connect them

Bitstream — binary file that configures the FPGA

Simulation Output

Waveforms — signal values over time

Text output — from $display

Pass/fail — from self-checking testbenches

VCD file — for GTKWave viewing

Synthesis produces physical outputs: a netlist of gates, a placement on the chip, routing between gates, and a bitstream binary. Simulation produces behavioral outputs: waveforms showing signal values over time, text from display statements, pass/fail from self-checking testbenches, and VCD files for waveform viewing.

The Synthesizable Subset

Not everything in Verilog can become hardware.

✓ Synthesizable

assign y = a & b;
always @(posedge clk)
    q <= d;
module, endmodule
wire, reg
if, case, for (fixed)

✗ Simulation Only

#10          // delay
$display()   // print
$dumpfile()  // waveform dump
$finish      // end sim
initial      // no hardware equiv
$readmemh()  // file I/O*
Rule: Design code uses only the synthesizable subset. Testbench code can use everything.
Not everything in Verilog becomes hardware. assign, always-posedge, modules — those synthesize into real gates. But delay statements, display, dumpfile, finish, initial blocks — these are simulation-only. The synthesizer ignores them entirely. Rule of thumb: your design code uses only the synthesizable subset. Your testbench code can use everything. We color-code these throughout the course: green border means synthesizable, orange means simulation-only.

The #10 Trap

// DANGER: this "works" in simulation...
always @(posedge clk) begin
    a <= 1'b1;
    #10;           // ← wait 10 time units
    a <= 1'b0;
end
Simulation: #10 pauses for 10 time units. It "works."
Synthesis: #10 is silently ignored. The behavior is completely different.
Result: Simulation passes, hardware fails. The worst kind of bug.
Never use #delay in synthesizable code. Not even "just for testing."
Here's a trap that catches people. This code uses hash-10 inside an always-posedge block. In simulation, it pauses for 10 time units and it appears to work correctly. In synthesis, hash-10 is silently ignored. The synthesized hardware behaves completely differently from the simulation. This is the worst kind of bug — one where simulation passes and hardware fails. Rule: never use hash-delay in synthesizable code. Not even "just for testing."

Your Day-to-Day Workflow

①  Write your Verilog module

②  Simulate it — does it behave correctly? (Icarus + GTKWave)

③  Synthesize it — does it fit? Does timing pass? (Yosys + nextpnr)

④  Program the FPGA — does it work on real hardware? (iceprog)

Always simulate first. Fixing bugs in simulation takes minutes. Debugging on hardware takes hours.
Here's your daily workflow for this course. Step 1: write your module. Step 2: simulate it — does it behave correctly? Step 3: synthesize it — does it fit on the FPGA? Step 4: program and test on hardware. Always simulate first. Fixing bugs in simulation takes minutes. Debugging on hardware takes hours. This workflow becomes second nature by the end of Week 1.

Key Takeaways

①  Same source file → two paths: synthesis (hardware) and simulation (behavior).

②  Not all Verilog synthesizes. #delay, $display, initial are sim-only.

③  Always simulate first, then synthesize to hardware.

Recap. One: same source, two paths — synthesis makes hardware, simulation tests behavior. Two: not all Verilog synthesizes. Hash-delay, dollar-display, initial are simulation only.
