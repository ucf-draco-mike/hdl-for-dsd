One Source File, Two Consumers

module.v — your Verilog source

⚙️ Synthesis

"Turn this into hardware"

Yosys → netlist of real gates

🔍 Simulation

"Show me how it behaves"

Icarus Verilog → waveforms

Both consume the same .v file. They do completely different things with it.
Your Verilog source file has two consumers. Synthesis turns it into hardware — actual gates mapped to an FPGA. Simulation shows you how the design behaves without touching hardware. Both consume the exact same dot-v file. They do completely different things with it.

The Open-Source iCE40 Toolchain

Here's our complete toolchain. On the left, the synthesis path: Yosys synthesizes your Verilog into a netlist of iCE40 primitives. nextpnr places and routes those primitives onto the physical FPGA. icepack generates the bitstream. iceprog loads it onto the board via USB. On the right, the simulation path: Icarus Verilog compiles your Verilog and testbench, vvp runs the simulation, and GTKWave lets you view the waveforms. Same source file, two completely different pipelines.

What Each Path Produces

Synthesis Output

Netlist — a circuit of real gates (LUTs, FFs)

Placement — which physical location on the chip

Routing — which wires connect them

Bitstream — binary file that configures the FPGA

Simulation Output

Waveforms — signal values over time

Text output — from $display

Pass/fail — from self-checking testbenches

VCD file — for GTKWave viewing

Synthesis produces physical outputs: a netlist of gates, a placement on the chip, routing between gates, and a bitstream binary. Simulation produces behavioral outputs: waveforms showing signal values over time, text from display statements, pass/fail from self-checking testbenches, and VCD files for waveform viewing.

The Synthesizable Subset

Not everything in Verilog can become hardware.

✓ Synthesizable

assign y = a & b;
always @(posedge clk)
    q <= d;
module, endmodule
wire, reg
if, case, for (fixed)

✗ Simulation Only

#10          // delay
$display()   // print
$dumpfile()  // waveform dump
$finish      // end sim
initial      // no hardware equiv
$readmemh()  // file I/O*
Rule: Design code uses only the synthesizable subset. Testbench code can use everything.
Not everything in Verilog becomes hardware. assign, always-posedge, modules — those synthesize into real gates. But delay statements, display, dumpfile, finish, initial blocks — these are simulation-only. The synthesizer ignores them entirely. Rule of thumb: your design code uses only the synthesizable subset. Your testbench code can use everything. We color-code these throughout the course: green border means synthesizable, orange means simulation-only.

The #10 Trap

// DANGER: this "works" in simulation...
always @(posedge clk) begin
    a <= 1'b1;
    #10;           // ← wait 10 time units
    a <= 1'b0;
end
Simulation: #10 pauses for 10 time units. It "works."
Synthesis: #10 is silently ignored. The behavior is completely different.
Result: Simulation passes, hardware fails. The worst kind of bug.
Never use #delay in synthesizable code. Not even "just for testing."
Here's a trap that catches people. This code uses hash-10 inside an always-posedge block. In simulation, it pauses for 10 time units and it appears to work correctly. In synthesis, hash-10 is silently ignored. The synthesized hardware behaves completely differently from the simulation. This is the worst kind of bug — one where simulation passes and hardware fails. Rule: never use hash-delay in synthesizable code. Not even "just for testing."

Your Day-to-Day Workflow

①  Write your Verilog module

②  Simulate it — does it behave correctly? (Icarus + GTKWave)

③  Synthesize it — does it fit? Does timing pass? (Yosys + nextpnr)

④  Program the FPGA — does it work on real hardware? (iceprog)

Always simulate first. Fixing bugs in simulation takes minutes. Debugging on hardware takes hours.
Here's your daily workflow for this course. Step 1: write your module. Step 2: simulate it — does it behave correctly? Step 3: synthesize it — does it fit on the FPGA? Step 4: program and test on hardware. Always simulate first. Fixing bugs in simulation takes minutes. Debugging on hardware takes hours. This workflow becomes second nature by the end of Week 1.

Your First Testbench

Step ② says "simulate it." But simulate it against what?

A testbench is a separate Verilog module that:

①  Has no ports — it's the top of the simulation world

②  Instantiates your design (the "DUT" — Design Under Test)

③  Drives inputs and checks outputs

④  Prints PASS / FAIL so you know immediately if something broke

You won't write testbenches from scratch until Week 2. For now you just need to read and run the ones we provide.
Step 2 says simulate it. But simulate it against what? A testbench. A testbench is a separate Verilog file — a module with no ports — that instantiates your design, drives its inputs, and checks its outputs. It prints PASS or FAIL so you know immediately whether your design is correct. You won't write testbenches from scratch until Week 2. For now, you just need to read and run the ones we provide.

Reading a Testbench

// tb_button_logic.v — tests your button_logic module
`timescale 1ns / 1ps            // ← time units for simulation

module tb_button_logic;          // ← no ports!
    reg  sw1, sw2, sw3, sw4;    // ← inputs you control
    wire led1, led2, led3, led4; // ← outputs to check

    button_logic dut (           // ← your design goes here
        .i_switch1(sw1), .i_switch2(sw2),
        .i_switch3(sw3), .i_switch4(sw4),
        .o_led1(led1), ...
    );

    initial begin
        $dumpfile("dump.vcd");   // ← record waveforms
        $dumpvars(0, tb_button_logic);

        sw1=0; sw2=1; sw3=0; sw4=1; // ← drive inputs
        #1;                          // ← wait for logic to settle
        if (led1 === 1'b0) ...       // ← check with ===
    end
endmodule
Notice: reg, initial, #1, $dumpfile — all sim-only constructs. A testbench will never become hardware.
Here's what a testbench looks like. This one tests the button-logic exercise you'll do today. At the top: timescale sets the time units. The module has no ports — it's the top level of simulation. Inside, reg signals are the inputs you control; wires are the outputs you check. The dut line instantiates your design — this is where your module plugs in. In the initial block: dumpfile and dumpvars record waveforms for GTKWave. Then you drive inputs, wait with hash-1 for combinational logic to settle, and check outputs with the triple-equals operator. Notice everything here — reg, initial, hash-delay, dollar-dumpfile — is simulation-only. A testbench never becomes hardware.

Running a Testbench

# From any lab exercise directory:
make sim       # compile + run → prints PASS/FAIL
make wave      # opens GTKWave to inspect waveforms

What make sim does under the hood:

iverilog -g2012 -Wall -o sim.vvp tb_button_logic.v button_logic.v
vvp sim.vvp

iverilog compiles Verilog → simulator binary.  vvp runs it → prints output + writes VCD.

PASS: sw=0000 -> led=0001
PASS: sw=0011 -> led=1100
...
=== tb_button_logic: 16 passed, 0 failed ===
ALL TESTS PASSED
Green text, zero failures → safe to synthesize.
Running a testbench is simple. From any lab directory, type make-sim. That compiles your design and testbench with Icarus Verilog, then runs the simulation. You'll see PASS or FAIL for each test case, and a summary at the bottom. If you want to see waveforms, type make-wave — that opens GTKWave with the recorded VCD file. Under the hood, make-sim runs two commands: iverilog to compile, and vvp to execute. The output looks like this — PASS for each test, and a final summary. Green text, zero failures — you're safe to synthesize. We'll dig much deeper into writing your own testbenches in Week 2. For now, just run the ones we provide and read the output.

Key Takeaways

①  Same source file → two paths: synthesis (hardware) and simulation (behavior).

②  Not all Verilog synthesizes. #delay, $display, initial are sim-only.

③  A testbench drives your design's inputs, checks its outputs, and prints PASS/FAIL.

④  Always simulate first — make sim before make prog.

Recap. One: same source, two paths — synthesis makes hardware, simulation tests behavior. Two: not all Verilog synthesizes. Hash-delay, dollar-display, initial are simulation only. Three: a testbench drives your design's inputs, checks its outputs, and prints PASS or FAIL — you'll run these from Day 1. Four: always simulate first. make-sim before make-prog.
