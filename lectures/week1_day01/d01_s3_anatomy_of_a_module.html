The Module: Verilog's Building Block

Every Verilog design is composed of modules.

Think of a module as a chip on a circuit board with labeled pins. You define what goes in, what comes out, and what happens inside.
Every Verilog design is composed of modules. A module is a self-contained hardware block with defined inputs and outputs. Think of it as a chip on a circuit board. You define what goes in — the input pins — what comes out — the output pins — and what happens inside — the behavior. This diagram shows a module called led_driver with inputs on the left, outputs on the right, and the assign statement that defines its behavior in the center.

⚠️ If You're Thinking Like a Programmer…

❌ Wrong Model

“input and output are like function parameters. Data ‘flows in’ through input ports when I ‘call’ the module.”

✓ Right Model

Module ports are physical pins on a chip-shaped abstraction. input ports are pins pulled by external wires; output ports are pins driven by internal logic. There's no “call” — the module is always active, like a DIP package soldered onto a board.

Consequence: There's no return value. Every value the module produces comes out of a named output port. Multi-output modules are the norm, not the exception.
The “module as chip” mental model is load-bearing for the whole course. When students later instantiate a module inside another module, they need to think of it as placing a chip on a board — named wires run between ports. The function-call metaphor fails badly at hierarchy (Topic 8).

The Module Template

module led_driver (
    input  wire i_switch1,
    output wire o_led1
);

    assign o_led1 = i_switch1;

endmodule

① module keyword + name

② Input ports — signals coming in

③ Output ports — signals going out

④ Semicolon after closing paren — don't forget this!

⑤ Behavior — assign, always, or sub-module instances

⑥ endmodule — no semicolon here

Here's the module template. We'll build it line by line. First: the module keyword and a name. Then input ports — signals coming in. Output ports — signals going out. A semicolon after the closing parenthesis — you WILL forget this and the error message will be unhelpful. Inside, the behavior — here it's a simple assign. And finally endmodule. Note: no semicolon after endmodule.

Port Style: ANSI (Modern) vs. Non-ANSI

✓ ANSI Style (use this)

module adder (
    input  wire [3:0] i_a,
    input  wire [3:0] i_b,
    output wire [4:0] o_sum
);
    assign o_sum = i_a + i_b;
endmodule

✗ Non-ANSI (legacy)

module adder (i_a, i_b, o_sum);
    input  [3:0] i_a;
    input  [3:0] i_b;
    output [4:0] o_sum;

    assign o_sum = i_a + i_b;
endmodule
ANSI style declares direction and type in one place. Less code, fewer bugs, easier to read. We use ANSI style exclusively in this course.
There are two port declaration styles. ANSI style, on the left, declares direction and type right in the port list. Non-ANSI, the legacy style, separates them — you list port names at the top, then declare them again below. ANSI style is less code, fewer bugs, easier to read. We use ANSI exclusively in this course. You may see non-ANSI in older textbooks or legacy code.

Naming Conventions

module my_design (
    input  wire       i_clk,        // i_ = input
    input  wire       i_reset,
    input  wire [7:0] i_data,
    output wire       o_valid,       // o_ = output
    output reg  [7:0] o_result
);
    wire [7:0] w_intermediate;       // w_ = internal wire
    reg  [3:0] r_counter;            // r_ = register (sequential)
    // ...

Naming conventions. We use a prefix system throughout this course. i-underscore for inputs, o-underscore for outputs, w-underscore for internal wires, r-underscore for registers. These are a convention, not a language requirement. But they're incredibly useful — you can tell at a glance what kind of signal you're looking at. Professional Verilog teams use similar conventions.

Common Error #1: The Missing Semicolon

module my_module (
    input  wire a,
    output wire y
)                         // ← MISSING SEMICOLON
    assign y = a;
endmodule
module my_module (
    input  wire a,
    output wire y
);                        // ← CORRECT
    assign y = a;
endmodule

You will make this mistake. The error message won't point to the right line. Now you know what to look for.

The number one syntax error you'll encounter: forgetting the semicolon after the closing parenthesis of the port list. The error message from iverilog or Yosys won't always point to the right line, which makes it hard to find. You will make this mistake. Now you know what to look for.

Common Error #2: Mismatched Widths

module oops (
    input  wire [7:0] i_data,    // 8 bits
    output wire [3:0] o_result   // 4 bits
);
    assign o_result = i_data;    // ← truncation! upper 4 bits lost
endmodule
Verilog silently truncates or zero-extends when widths don't match. No error — just quiet data loss. We'll cover this in detail on Topic 2.
Second common error: mismatched widths. Here, an 8-bit input is assigned to a 4-bit output. Verilog silently truncates — the upper 4 bits just vanish. No error, no warning by default. Just quiet data loss. Width mismatches are a major source of bugs. We'll cover this properly on Topic 2.

Ask an AI assistant: “What happens if I declare a signal as output reg but then drive it with an assign statement outside the module?”

TASK

Ask about mixing output reg with external assign.

BEFORE

Predict: “Compile error”? “Silent bug”? What's the failure mode?

AFTER

Correct answer: external assign to an output reg is a Verilog error. iverilog catches this.

TAKEAWAY

Compile first, trust the compiler second, verify third. Use iverilog -Wall always.

Key habit to build early: use -Wall. Verilog's compile warnings catch a surprising number of design bugs if you actually read them.

Key Takeaways

①  Modules are the fundamental building block — inputs, outputs, behavior.

②  assign = permanent wire, not a one-time computation.

③  Use ANSI-style ports and consistent naming (i_, o_, r_, w_).

④  Don't forget the semicolon after ); in the port list.

Recap. One: modules are the fundamental building block with inputs, outputs, and behavior. Two: assign creates a permanent wire — not a one-time computation.
