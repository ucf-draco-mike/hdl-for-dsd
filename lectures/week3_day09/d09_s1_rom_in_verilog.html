ROM in an FPGA

No physical ROM chip — ROM is implemented as:

LUT-based: Small ROMs packed into lookup tables (same ones used for logic)

Block RAM-based: Larger ROMs mapped to dedicated EBR blocks, initialized at config

You write the same Verilog regardless. The synthesis tool decides placement based on size and access pattern.
FPGA ROM is implemented in LUTs or block RAM. You write one style of Verilog; the tools choose the resource.

Approach 1: case-Based ROM

module rom_case (
    input  wire [3:0] i_addr,
    output reg  [7:0] o_data
);
    always @(*) begin
        case (i_addr)
            4'h0: o_data = 8'h3E;
            4'h1: o_data = 8'h41;
            4'h2: o_data = 8'h49;
            // ... 16 entries total ...
            default: o_data = 8'h00;
        endcase
    end
endmodule
Doesn’t scale. 256 entries = 256 case items. Use for tiny hand-crafted tables only.
Case-based ROM is readable for small tables but doesn't scale. 256 entries becomes unmanageable.

Approach 2: Array + $readmemh

module rom_array #(
    parameter ADDR_W = 8,
    parameter DATA_W = 8,
    parameter MEM_FILE = "rom_data.hex"
)(
    input  wire [ADDR_W-1:0] i_addr,
    output wire [DATA_W-1:0] o_data
);
    reg [DATA_W-1:0] r_mem [0:(1<<ADDR_W)-1];

    initial $readmemh(MEM_FILE, r_mem);

    assign o_data = r_mem[i_addr];  // async read → LUT-based
endmodule
$readmemh works in both simulation (Icarus) and synthesis (Yosys). Change the file, change the ROM — no recompilation.
Array-based ROM with readmemh scales to any size. Works in both simulation and synthesis. Change the hex file, change the behavior.

Key Takeaways

① case-ROM: small, hand-crafted. Array-ROM: scalable, file-driven.

② $readmemh works in simulation and synthesis.

③ Async read (assign) → LUT-based. Sync read → block RAM.

④ Change the .hex file to change behavior — no recompile needed.
