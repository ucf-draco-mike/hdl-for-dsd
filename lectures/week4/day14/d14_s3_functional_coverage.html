The Coverage Question

Assertions check correctness. Coverage measures completeness.

Code coverage: “Did every line execute?” — necessary but not sufficient.

Functional coverage: “Did I test interesting combinations?” — you define what matters.

100% code coverage ≠ 100% bug-free. But 60% functional coverage = 40% of interesting scenarios untested.
Assertions check correctness. Coverage measures completeness. Code coverage is necessary but not sufficient. Functional coverage lets you define what 'interesting' means.

covergroup Example

covergroup alu_cov @(posedge clk);
    cp_op: coverpoint opcode {
        bins add = {2'b00}; bins sub = {2'b01};
        bins bw_and = {2'b10}; bins bw_or = {2'b11};
    }
    cp_a: coverpoint a {
        bins zero = {4'h0};
        bins mid  = {[4'h1:4'hE]};
        bins max  = {4'hF};
    }
    // Cross: every opcode × every operand range
    cx_full: cross cp_op, cp_a;
endgroup
Run your tests → check coverage report → add tests for gaps → repeat. Coverage-driven workflow.
Covergroup defines what to measure. Coverpoints are individual signals with bins. Cross coverage checks combinations. Run tests, find gaps, add tests, repeat.

Key Takeaways

① Assertions = correctness. Coverage = completeness.

② covergroup + coverpoint + bins = measurement framework.

③ cross checks interesting combinations automatically.

④ Coverage-driven: define → test → measure → fill gaps → repeat.
