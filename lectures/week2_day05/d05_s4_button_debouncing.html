The Bounce Problem

Mechanical switches don’t make clean transitions — they bounce for 1–20 ms.

At 25 MHz, a 5 ms bounce window = 125,000 false edges. Your counter increments 125,000 times per press!
Mechanical switches bounce for 1-20 ms. At 25 MHz, that's up to 500,000 false edges. Your counter goes haywire.

Counter-Based Debouncer

module debounce #(
    parameter CLKS_TO_STABLE = 250_000  // 10ms at 25MHz
)(
    input  wire i_clk, i_bouncy,
    output reg  o_clean
);
    reg [$clog2(CLKS_TO_STABLE)-1:0] r_count;
    reg r_sync_0, r_sync_1;  // 2-FF synchronizer

    always @(posedge i_clk) begin
        r_sync_0 <= i_bouncy;
        r_sync_1 <= r_sync_0;  // synchronized input
    end

    always @(posedge i_clk) begin
        if (r_sync_1 != o_clean) begin
            r_count <= r_count + 1;
            if (r_count == CLKS_TO_STABLE - 1)
                o_clean <= r_sync_1;
        end else
            r_count <= 0;
    end
endmodule
Counter-based debounce: synchronize first, then count how long the input stays different from the output. If it persists for CLKS_TO_STABLE cycles, accept the new value. Bounces reset the counter.

The Input Pipeline

Button (async) → [2-FF Sync] → [Debounce Counter] → [Edge Detect] → Clean pulse
                  ↑ solves          ↑ solves            ↑ creates
              metastability      bounce noise        one-cycle pulse
    
// Edge detector: one-clock-cycle pulse on rising edge
reg r_clean_prev;
always @(posedge i_clk) r_clean_prev <= o_clean;
wire w_press_edge = o_clean & ~r_clean_prev;
This pipeline — sync → debounce → edge detect — is the standard pattern for all button inputs. Build it once, reuse everywhere.

Pre-Class Self-Check

Q1: What does $clog2(N) return and why is it useful?

ceil(log₂(N)) — the number of bits needed for values 0 to N−1. Auto-sizes counter widths when N changes.

Q2: What type of shift register is the core of a UART TX? RX?

TX = PISO (load byte, shift out). RX = SIPO (shift bits in, present byte).

Pre-Class Self-Check (cont.)

Q3: What causes metastability and what’s the standard fix?

Async signal violates setup/hold timing. Fix: 2-FF synchronizer — first FF may go metastable but has a full period to resolve before the second samples it.

Q4: Why synchronize before debouncing?

The debounce counter is clocked logic. If its input is async, its own FFs can go metastable. Synchronize first to make the signal safe for all downstream logic.
