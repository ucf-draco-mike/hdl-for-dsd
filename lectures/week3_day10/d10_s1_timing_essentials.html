Why Timing Matters

Every gate and wire has propagation delay. Signals take time to travel through combinational logic between flip-flops.

The critical path determines the maximum clock frequency of your design.
Every gate and wire has delay. The longest path between any two flip-flops is the critical path — it sets your maximum frequency.

Setup & Hold Time

               Tsetup      Thold
              ◄──────►    ◄──────►
              │        │          │
   data: ─────────────STABLE───────────────
              │        │          │
   clk:  ────────────┐ │
                      └───────────────
                      ↑
                  clock edge

Setup violation: data changes too close before the edge → wrong value captured

Hold violation: data changes too soon after the edge → wrong value captured

Slack = clock_period − (Tclk-to-q + Tcomb + Trouting + Tsetup). Positive = timing met. Negative = violation.
Setup: data stable before edge. Hold: data stable after. Slack is your margin. At 25 MHz with 40 ns period, most simple designs pass easily. Problems appear with deep combinational chains or higher clocks.

Reading the nextpnr Timing Report

# Tell nextpnr about your clock:
nextpnr-ice40 --hx1k --package vq100 \
    --pcf go_board.pcf --json top.json --asc top.asc \
    --freq 25

# PASS — Fmax exceeds target
Info: Max frequency for clock 'i_clk': 48.23 MHz
#  Target: 25 MHz → PASS (23 MHz of margin)

# FAIL — Fmax below target
Info: Max frequency for clock 'i_clk': 22.10 MHz
#  Target: 25 MHz → FAIL! Design may work intermittently.
Timing violations cause intermittent failures — the worst kind of bug. The design works sometimes, fails randomly.
Always add --freq to nextpnr. Without it, no timing analysis happens. Above target = pass. Below = violation. Violations cause intermittent failures — maddening to debug.

When Timing Fails: Three Fixes

① Pipeline: insert registers to break long combinational chains. Costs latency + FFs.

② Restructure logic: rethink the architecture. Sequential multiplier instead of combinational?

③ Relax the clock: lower the frequency if the application allows it.

Three options: pipeline (trade latency for throughput), restructure (change the architecture), or slow the clock. We'll see concrete examples of options 1 and 2 in the next segment when we compare adder and multiplier architectures.

Key Takeaways

① Critical path = longest FF-to-FF delay → limits maximum frequency.

② Slack = clock_period − total_delay. Positive = pass. Negative = intermittent failures.

③ --freq 25 on nextpnr enables timing analysis. Always include it.

④ Fixes: pipeline, restructure, or relax the clock — all have PPA trade-offs.
