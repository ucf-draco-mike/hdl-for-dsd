This is the core of Day 10. We're going to look at what the synthesis tool actually builds when you write arithmetic in Verilog, and why understanding that matters for design decisions.

What Does + Actually Build?

// You write this:
assign sum = a + b;

// Yosys builds... what?

Key insight: Writing assign sum = a + b; lets the synthesis tool choose the adder architecture.

Understanding what it chooses — and why — is the designer's job.

Today we'll synthesize a + b at 4, 8, 16, and 32 bits — and inspect the results with yosys stat and yosys show.
When you write a+b, you're delegating the architecture choice to the tool. That's fine — but you need to understand what it builds so you can evaluate whether it's appropriate for your application.

Adder Architectures

Ripple-carry (Day 2): carry propagates bit-by-bit. Simple. Slow for wide operands.
Delay: O(N). Area: O(N). You built this from full-adder chains.

Carry-lookahead (concept): precompute carries in parallel. Faster. More logic.
Delay: O(log N). Area: O(N log N). Used in high-performance designs.

The + operator: Yosys picks the best architecture for the target FPGA.
On iCE40: typically a carry-chain structure using dedicated routing.

Lab exercise: You'll synthesize your Day 2 ripple-carry adder and assign sum = a + b; at 8-bit and 16-bit, then compare LUT counts and Fmax.
Three architectures at increasing sophistication. The key learning: the behavioral + operator often matches or beats your manual implementation, because the tool knows the FPGA's carry chain resources. But understanding WHY is the point.

Multiplication: The LUT Explosion

// Combinational multiplication
assign product = a * b;  // 4-bit: manageable. 8-bit: large. 16-bit: enormous.

Why it explodes: Multiplication is O(N²) in gate count. Each partial product needs a row of AND gates, then all rows are summed.

iCE40 has NO DSP blocks. Every * is pure LUT logic. A 32-bit multiplier can consume ~30% of the HX1K.

On FPGAs with DSP slices (Xilinx, larger Lattice), * maps to dedicated silicon — nearly free. On iCE40, it's expensive.
This is the biggest resource trap on iCE40. No DSP blocks means every multiply is built from LUTs. At 8-bit you'll see quadratic growth. In lab you'll measure this directly.

Sequential Multiplication: Shift-and-Add

Algorithm:

1. Initialize accumulator = 0

2. For each bit of the multiplier:

If bit is 1: add multiplicand to accumulator

Shift multiplicand left (or shift accumulator right)

3. After N iterations: accumulator = product

Architecture:

FSM + shift register + accumulator

Trade-off:

Comb *Shift-Add
Latency1 cycleN cycles
AreaO(N²) LUTsO(N) LUTs
FmaxLowerHigher
The shift-and-add multiplier trades latency for area. Instead of building all partial products simultaneously, it does one per clock cycle. Uses an FSM, a shift register, and an accumulator — all things you built in Weeks 1 and 2. This is your lab Exercise 2.

Fixed-Point Arithmetic: Q-Format

// Q4.4 format: 4 integer bits, 4 fractional bits = 8 bits total
// Value = binary_integer_interpretation / 2^(fractional_bits)
// Example: 8'b0010_1000 = 0x28 = 40 → 40/16 = 2.5

wire [7:0] a = 8'b0010_1000;  // 2.5 in Q4.4
wire [7:0] b = 8'b0011_0000;  // 3.0 in Q4.4

Addition: Just works — assign sum = a + b; (same format, watch for overflow)

Multiplication — the bit growth problem:

// Q4.4 × Q4.4 = Q8.8 (16 bits!)
wire [15:0] full_product = a * b;       // Q8.8
wire [7:0]  q44_result = full_product[11:4]; // Extract Q4.4 from Q8.8
Key challenge: Know where the decimal point is after each operation. The hardware doesn't track it — you do.
Fixed-point is everywhere in DSP and control systems. The hardware is just integer arithmetic — YOU track the decimal point. The biggest gotcha: multiplying two Q4.4 numbers gives Q8.8. You need to extract the right bits.

Signed Arithmetic Gotchas

// Verilog defaults to UNSIGNED arithmetic!
wire [7:0] a = 8'hFF;  // Unsigned: 255.  Signed: -1.

// To get signed behavior:
wire signed [7:0] a_signed = 8'hFF;  // Now it's -1

// Or use $signed() cast:
wire [15:0] product = $signed(a) * $signed(b);

// Sign extension matters for mixed-width operations:
wire [15:0] extended = {{8{a[7]}}, a};  // Manual sign-extend
wire [15:0] extended = $signed(a);       // Automatic sign-extend
Common bug: Mixing signed and unsigned in one expression. Verilog silently treats the whole expression as unsigned if any operand is unsigned.
Verilog defaults to unsigned. If you need signed arithmetic, use the signed keyword or $signed() cast. The sneaky bug: mixing signed and unsigned makes the whole expression unsigned. Be explicit.

Key Takeaways

① + lets the tool choose the adder architecture. Understand what it builds.

② * on iCE40 = pure LUT logic (no DSP). Area grows O(N²). Budget carefully.

③ Shift-and-add: trades N cycles of latency for O(N) area. Use when area-constrained.

④ Fixed-point: hardware is just integer arithmetic — you track the decimal point.

⑤ $signed() for signed operations. Don't mix signed/unsigned.
