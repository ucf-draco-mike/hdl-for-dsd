The Problem: A Two-Stage Pipeline

// WRONG: blocking in sequential
always @(posedge clk) begin
    b = a;    // b gets a's value NOW
    c = b;    // c gets b's NEW value (= a!) — both same!
end
// CORRECT: nonblocking in sequential
always @(posedge clk) begin
    b <= a;   // scheduled: b ← a(current)
    c <= b;   // scheduled: c ← b(current)
end           // at end: b=a(old), c=b(old) — proper pipeline!
With blocking: c gets the new value of b, which is already a. Both end up the same — pipeline is destroyed. With nonblocking: both see pre-update values, proper pipeline behavior.

Nonblocking: How It Works

Step 1: Evaluate ALL right-hand sides using current values

Step 2: Schedule all updates (don’t apply yet)

Step 3: Apply all updates simultaneously at end of timestep

This models real hardware: all flip-flops capture their D inputs simultaneously on the clock edge. No flip-flop “sees” another’s new value.
Three steps: evaluate all RHS, schedule all updates, apply simultaneously. This models real hardware where all FFs capture simultaneously.

The Golden Rules

=  in  always @(*)  — combinational

<=  in  always @(posedge clk)  — sequential

Never mix. Never break this rule.
Memorize this. Blocking for combinational. Nonblocking for sequential. Never mix. This rule will serve you for your entire hardware design career.

Key Takeaways

① <= evaluates all RHS first, then updates simultaneously.

② This models real flip-flop behavior — simultaneous capture.

③ = in sequential blocks destroys pipeline behavior.

④ The rule is absolute: = for @(*), <= for @(posedge).
