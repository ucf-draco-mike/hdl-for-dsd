The Methodology

1. Identify states — what distinct modes does the system have?

2. Define transitions — what input conditions cause state changes?

3. Assign outputs — what does each state produce?

4. Draw the state diagram — on paper, always!

5. Code it — translate diagram to 3-block template

6. Test it — verify every transition in simulation

The systematic approach: identify states, define transitions, assign outputs, draw the diagram ON PAPER, code it using the template, test every transition.

FSM Coding Checklist

☑ Block 1: Only r_state <= r_next_state (no logic!)

☑ Block 2: Default r_next_state = r_state (first line)

☑ Block 2: default case → safe state

☑ Block 3: Default outputs (first lines)

☑ Block 3: default case → safe outputs

☑ = in Blocks 2 & 3, <= in Block 1

☑ Testbench covers every transition + reset

The FSM checklist. Go through this every time you write an FSM. Block 1: trivial. Block 2: default first, default case. Block 3: output defaults, default case. Correct assignment types. Test everything.

Pre-Class Self-Check

Q1: What are the three blocks in the 3-always-block style?

1. State Register (sequential, just a FF). 2. Next-State Logic (combinational). 3. Output Logic (combinational — Moore: f(state) only).

Q2: What is the critical first line in Block 2 and why?

r_next_state = r_state; — prevents latch inference and means "stay in current state by default."

Pre-Class Self-Check (cont.)

Q3: Moore vs. Mealy — which do we default to and why?

Moore — outputs depend only on state. Safer, easier to debug, cleaner timing. Mealy only for same-cycle reaction.

Q4: Name three common FSM coding mistakes.

1. No default in Block 2 → latch. 2. No output defaults in Block 3 → latches. 3. No default case → stuck in illegal state. 4. Using = in Block 1 (should be <=).
